# DevNest E2Eシナリオ & ユーザーシナリオ網羅マトリクス — Phase 6〜10 補遺

> **目的**: Phase 6-10テスト計画の補遺として、ユーザージャーニー起点のE2Eシナリオと
> 全機能のユーザーシナリオ網羅表を定義する。
> **関連**: `devnest-phase6-10-test-plan.md` のE2Eセクションを拡張する位置づけ。
> **重要**: 既存のユーザージャーニー（J-00〜J-07）およびシナリオ（S-01〜S-18）と
> Phase 6-10の新機能を統合し、抜け漏れを防ぐ。

---

## 1. ユーザーペルソナ定義

Phase 6-10で想定する利用者の類型。

| ペルソナ | 説明 | 主な利用Phase |
|---------|------|-------------|
| **Solo Dev（個人開発者）** | 個人プロジェクトを複数管理。DevNest＋Claude Codeで開発 | 6, 7, 8 |
| **Work Dev（職場開発者）** | 職場のプロジェクトをDevNestで管理。Redmine/Slack連携 | 7, 8, 9 |
| **Team Lead（チームリード）** | チームの生産性を可視化し、プロセス改善を推進 | 7, 8, 10 |
| **New User（新規ユーザー）** | DevNestを初めて使い始める。セットアップから始める | 6-10（初回フロー） |

## 2. ユーザージャーニーマップ（Phase 6-10拡張）

既存J-00〜J-07に加え、Phase 6-10で追加されるジャーニーを定義。

### J-08: AIアシスタントで開発を加速する

```
ペルソナ: Solo Dev
ゴール: 設計書を元にAIでコードを生成し、レビューを受けて実装を完了する

フロー:
1. DevNestで設計書を開く
2. 「この設計書からコード生成」を実行
3. 生成されたコード骨格を確認
4. 不足部分をClaude Codeで実装補完
5. AIレビューを実行
6. 設計書との整合性チェック結果を確認
7. 指摘事項を修正
8. PRを作成、doc-mappingが自動更新される

関連Phase: 6
関連タスク: 6.1, 6.2, 6.3, 6.4
```

### J-09: 開発のインパクトを振り返る

```
ペルソナ: Solo Dev / Work Dev
ゴール: スプリントの成果とAI活用効果を定量的に把握する

フロー:
1. 分析ダッシュボードを開く
2. 開発速度トレンドを確認（コミット数、LOC推移）
3. AI効果サマリーを確認（節約時間、採用率）
4. 前スプリントとの比較を確認
5. 改善すべきポイントを特定
6. 次スプリントのアクションに反映

関連Phase: 7
関連タスク: 7.1, 7.2, 7.3, 7.4
```

### J-10: スプリントを回して成長を積み重ねる

```
ペルソナ: Solo Dev
ゴール: 個人開発でもスプリントリズムを作り、継続的に改善する

フロー:
1. バックログにカードを整理
2. AIスプリントプランナーでプラン提案を受ける
3. プランを調整・承認してスプリント開始
4. カンバンでタスクを進行（WIP制限意識）
5. スプリント完了時にレトロスペクティブ自動生成
6. 年輪レトロスペクティブで成長を確認
7. アクションアイテムが次スプリントのBacklogに追加される

関連Phase: 8
関連タスク: 8.1, 8.2, 8.3, 8.5
```

### J-11: 外部ツールをDevNestに集約する

```
ペルソナ: Work Dev
ゴール: GitHub/Slack/RedmineをDevNestから統一的に操作する

フロー:
1. MCP接続管理画面でサーバーを設定
2. GitHub MCPに接続してツール一覧を確認
3. Slack MCPに接続して通知チャンネルを設定
4. Redmine MCPに接続してプロジェクトを選択
5. ポリシーを設定（職場プロジェクトは承認必須）
6. Agentic Flowから保守スキャン→Issue作成→Slack通知が自動で流れる
7. カンバンカードとRedmineチケットが同期される

関連Phase: 9
関連タスク: 9.1〜9.7
```

### J-12: チームの知見を蓄積・共有する

```
ペルソナ: Team Lead
ゴール: チームの開発状況を把握し、知見を組織に蓄積する

フロー:
1. チームダッシュボードでメンバー活動を確認
2. レビュー待ちPRを把握して声掛け
3. チーム全体のベロシティトレンドを確認
4. レトロスペクティブの学びをナレッジとして登録
5. 設計判断の記録をナレッジに追加
6. チームの年輪レトロスペクティブで成長を俯瞰
7. scrum-agentsでセレモニーを自動化

関連Phase: 10
関連タスク: 10.1, 10.2, 10.3
```

## 3. E2Eシナリオ（網羅版）

### カテゴリA: コア開発フロー

```
E2E-A1: 設計書→コード生成→レビュー→マージ（ハッピーパス）
─────────────────────────────────────
前提: TestRepoにAPI定義設計書が存在
ペルソナ: Solo Dev
ジャーニー: J-08

ステップ:
  1. 設計書を開き「コード生成」を実行
  2. Scaffold モードで Rust 関数シグネチャが生成される
  3. 生成ファイルが正しいディレクトリに配置される
  4. doc-mapping の mapping.sources に自動追加される
  5. AIレビューを実行
  6. 設計書との整合性チェックが PASS
  7. PR を作成（MockMcpServer経由）
  8. PRコメントにdoc-impact分析が含まれる
  9. 全設計書の status が current

検証ポイント:
  - 生成コードがコンパイル可能
  - frontmatter の version が更新されている
  - .doc-map.yaml が最新状態


E2E-A2: 設計書→コード生成で整合性エラー検出
─────────────────────────────────────
前提: 設計書のAPI定義と既存コードに不一致がある
ペルソナ: Solo Dev
ジャーニー: J-08

ステップ:
  1. 既存コードと矛盾する設計書変更をコミット
  2. AIレビューを実行
  3. DesignConsistencyReport に inconsistencies が検出される
  4. suggested_doc_updates に修正提案が含まれる
  5. overall_assessment が RequestChanges

検証ポイント:
  - 不一致箇所のファイルと行番号が正確
  - 修正提案が具体的


E2E-A3: テスト生成→カバレッジ改善
─────────────────────────────────────
前提: カバレッジが低いファイルが存在
ペルソナ: Solo Dev
ジャーニー: J-08

ステップ:
  1. カバレッジスキャンでホットパス（低カバレッジ×高churn）を特定
  2. 対象ファイルに対して TestOnly モードでコード生成
  3. 生成されたテストが実行可能
  4. カバレッジが改善される

検証ポイント:
  - 生成テストが cargo test で通る
  - カバレッジ数値が向上
```

```
E2E-B2: AI効果ゼロからの計測開始

前提: Agentic Flow を初めて有効化した直後
ペルソナ: New User

  - データ0件で安全に表示

E2E-B3: ポートフォリオ横断分析
─────────────────────────────────────

ペルソナ: Solo Dev

  4. プロダクト間のカバレッジ比較が表示される

検証ポイント:
  - プロダクト別データが混ざらない
  - プロダクト切替でURL/状態が正しく変わる
```

### カテゴリC: アジャイルエンジンフロー

```
E2E-C1: スプリントフルライフサイクル
─────────────────────────────────────
前提: バックログにカード5件、過去2スプリントのデータあり
ペルソナ: Solo Dev
ジャーニー: J-10

ステップ:
  1. AIスプリントプランナーでプラン提案を受ける
  2. 3件のカード + 1件の保守タスクが提案される
  3. プランを承認 → カードが InProgress に移動
  4. 保守タスクが Agentic Flow キューに追加される
  5. カードを1件ずつ Done に移動
  6. WIP制限（2）を超えようとすると警告
  7. 全カード完了
  8. レトロスペクティブ自動生成
  9. went_well に「全カード完了」が含まれる
  10. 年輪エントリが追加される
  11. アクションアイテムが次スプリントの Backlog に追加

検証ポイント:
  - プランのestimated_velocityが過去データに基づく
  - WIP制限が正しく機能
  - レトロの evidence に具体的数値
  - 年輪が蓄積される


E2E-C2: 空のバックログからのスプリント開始
─────────────────────────────────────
前提: バックログが空、保守データのみ存在
ペルソナ: New User
ジャーニー: J-10

ステップ:
  1. スプリントプランナーを実行
  2. 「カードがありません」+ 保守タスクのみのプランが提案
  3. 保守タスクを承認
  4. 技術的負債のインポートを実行
  5. 負債アイテムがBacklogにカードとして追加される
  6. 次回プランニングでカードが利用可能に

検証ポイント:
  - 空バックログでクラッシュしない
  - 負債→カード変換が正しい


E2E-C3: GitHub Issue同期付きカンバン運用
─────────────────────────────────────
前提: GitHub Issueが5件、MCP接続済み
ペルソナ: Work Dev
ジャーニー: J-10 + J-11

ステップ:
  1. sync_github_issues を実行
  2. 5件のIssueがカンバンカードに変換される
  3. カードを InProgress に移動
  4. GitHub Issue にラベル「in-progress」が追加される（MockMcpServer検証）
  5. カードを Done に移動
  6. GitHub Issue がクローズされる（MockMcpServer検証）
  7. MCP接続が切れた場合
  8. ローカル操作は継続可能
  9. 再接続時に差分同期

検証ポイント:
  - 双方向同期の整合性
  - オフライン耐性


E2E-C4: フロー最適化とWIP調整
─────────────────────────────────────
前提: 3スプリント分のカンバンフローデータ
ペルソナ: Solo Dev
ジャーニー: J-10

ステップ:
  1. フロー分析を実行
  2. Review列がボトルネックとして検出される
  3. WIP制限の最適化提案が表示される
  4. 提案を適用（Review列のWIPを3→2に）
  5. サイクルタイムのトレンドが表示される

検証ポイント:
  - ボトルネック検出ロジックの正確性
  - WIP変更後のカード移動制限が即座に反映


E2E-C5: ストーリーマッピングからカンバンへの連携
─────────────────────────────────────
前提: ストーリーマップにRelease 1, 2が定義されている
ペルソナ: Solo Dev
ジャーニー: J-10

ステップ:
  1. ストーリーマップで Release 1 のストーリーを選択
  2. 選択ストーリーをカンバンカードに変換
  3. カンバンの Backlog に追加される
  4. カードからストーリーマップへのリンクが有効
  5. ストーリーに設計書リンクを追加
  6. 設計書からストーリーへの逆参照が可能

検証ポイント:
  - ストーリー→カード変換の整合性
  - 双方向リンクの有効性
```
